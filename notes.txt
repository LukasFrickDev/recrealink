ONBOARDING DE CADASTRO PARA PRIMEIRO ACESSO POR MODAL - FUTURAMENTE


---------- OBJETIVO DA FASE ATUAL-------------
O objetivo agora não é “terminar o front” nem “começar o backend correndo”.

O objetivo é:

arrumar a estrutura do front atual para que ele represente corretamente o MVP, com páginas/módulos/coerência de dados suficientes para depois conectar backend sem retrabalho.

Em linguagem prática:

limpar excesso
alinhar páginas ao MVP
separar o que é núcleo, complementar e futuro
preparar contratos de dados/mock
organizar o módulo antes de implementar lógica real
3. Fases que eu seguiria a partir de agora
Fase A — Auditoria estrutural do front atual

Começar por um módulo de cada vez, sem mexer em tudo ao mesmo tempo.

Ordem recomendada:

Recreador
Empresa Hotelaria
Empresa Eventos
Pais apenas como referência visual

O repo já tem uma base compartilhada e uma regra clara de reutilizar o que for comum em src/shared, mantendo em cada módulo só o que for específico. Então essa auditoria deve aproveitar essa fundação, não recriar tudo de novo.

Saída dessa fase

lista de páginas que ficam
páginas que saem do núcleo
páginas que viram futura
componentes compartilháveis
contratos de dados/mock
dependências futuras com backend
ordem de implementação por módulo
Fase B — Congelamento estrutural da Etapa 1

Depois da auditoria, você congela:

estrutura de módulos
sitemap real da versão atual
páginas núcleo
contratos mockados principais
estados principais de cada fluxo

Aqui você para de polir detalhe visual sem impacto.

Fase C — Descida para arquitetura funcional

Só depois disso:

entidades principais
regras de negócio
auth e permissões
modelo de eventos/vagas/candidaturas/convites/bloqueios
definição de serviços e APIs
Fase D — Testes do núcleo

Antes de sair ligando tudo:

testes de regras de negócio
testes de conflito
testes de aceite/convite/candidatura
testes de auth/perfis

O Akita bate muito nessa tecla: sem testes úteis, cada evolução com IA vira roleta russa; com testes, você consegue refatorar e iterar em small releases com mais segurança.

Fase E — Redux/estado global

Redux só entra onde fizer sentido real:

sessão/auth
notificações globais
estados compartilhados entre módulos
filtros persistentes se realmente forem globais

Não vale “colocar Redux em tudo” só porque está na stack. O seu próprio repo já usa Redux Toolkit, mas a decisão agora deve ser de necessidade, não de obrigação.

Fase F — Primeiro fluxo vertical funcional

Primeiro fluxo real recomendado:
empresa cria evento → publica vaga → recreador vê → candidatura/convite → aceite → bloqueio automático

Esse fluxo é basicamente a definição do MVP validado.
